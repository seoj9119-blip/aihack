# PM / 기획 담당 실행 계획 (v2 — 4시간 스프린트 재계획)

> 기준 문서: [PRD_simple.md](./PRD_simple.md), [ROLE_ASSIGNMENT.md](./ROLE_ASSIGNMENT.md)
> **업데이트:** 실제 개발은 지금 막 시작되었고, 남은 시간은 총 **4시간**입니다. 4개 역할(PM, 프론트엔드, AI/백엔드, UI/UX·QA)이 각 1명씩 이미 병렬로 개발 중이라는 전제로 v1(24시간 계획)을 전면 재조정했습니다.
> 담당 역할: PRD 기반 전체 일정 관리 / 사용자 스토리 정리 및 우선순위 결정 / 팀 커뮤니케이션·데모 준비·리스크 관리

---

## 0. 지금 당장 결정해야 할 것 (선택)

시간이 24h → 4h로 줄면서 리스크 항목인 **F-03 드래그 UI**에 쓸 수 있는 시간이 크게 부족합니다. PM으로서 아래 결정을 팀과 지금 확인하는 것을 권장합니다.

- **F-03 구현 난이도를 처음부터 낮추기:** "문단 드래그 선택"을 밑바닥부터 만들지 말고, 처음부터 **"문단 클릭 → 선택 → 수정 버튼"** 형태의 단순 대체안으로 시작할지 결정 (아래 3장 참고)
- **API Key 처리 방식:** 프론트엔드 직접 호출 + 환경변수(.env, 데모용) vs 서버리스 함수 — 4시간 안에는 환경변수 방식이 현실적일 가능성이 높음

## 1. 전체 일정 (4시간 타임라인)

이미 4명이 동시에 개발 중이므로, v1의 "AI Core First(순차)" 전략 대신 **역할별 병렬 개발 + 짧은 통합 지점**으로 재구성했습니다.

| 시간대 | 단계 | 내용 | PM 체크포인트 |
|---|---|---|---|
| H0:00–H0:15 | 킥오프 동기화 | 위 0장 결정사항 확정, 폼↔AI 데이터 형식(어떤 필드를 어떤 키로 주고받을지) 합의 | API Key 방식 + F-03 대체안 여부 **지금 확정** |
| H0:15–H1:30 | 병렬 코어 개발 | AI/백엔드: F-02 프롬프트·API 로직 · 프론트엔드: F-01 템플릿 폼 UI · UI/UX: F-03 에디터 라이브러리(TipTap/Quill) 세팅 및 스파이크 | H1:30에 3개 조각이 각자 독립적으로 동작하는지 확인 |
| H1:30–H2:30 | 통합 | 폼 → AI 호출 → 결과 표시 엔드투엔드 연결, F-03 실제 데이터에 연동 | **1차 통합 데모** 가능 여부 확인 — 안 되면 F-03 즉시 단순 대체안으로 전환 |
| H2:30–H3:00 | 마무리 기능 + 기능 동결 | F-04 (복사/다운로드) 최소 구현 | **기능 동결(Feature Freeze)** — 이후 신규 기능 추가 금지 |
| H3:00–H3:30 | QA 버그픽스 | US-01~03 시나리오 기준 전체 흐름 점검, P1 버그 우선 배분 | 발견된 버그를 심각도순으로 개발 담당에 즉시 전달 |
| H3:30–H3:50 | 데모 준비 | 시연 시나리오 확정, 입력 데이터 준비, 리허설 1회 | 아래 4장 체크리스트 확인 |
| H3:50–H4:00 | 최종 점검 | 제출, 백업, 발표 | 제출 형식/마감시간 재확인 |

## 2. 사용자 스토리 & 기능 우선순위

PRD 4장(US-01~03), 5장(F-01~04) 기준. 시간이 6배 줄었으므로 **Cut line을 더 이르게, 더 공격적으로** 적용합니다.

| 우선순위 | 기능 | 연결 유저스토리 | 비고 |
|---|---|---|---|
| P1 (필수) | F-02 AI 기반 통합 초안 생성 | US-01, US-02 | 서비스 핵심가치. 절대 빠지면 안 됨 |
| P1 (필수) | F-01 목적별 템플릿 입력 폼 | US-01, US-02 | F-02 없이는 무의미하므로 세트로 진행 |
| P1 (필수, 축소 가능) | F-03 드래그 기반 부분 수정 | US-03 | **처음부터 "클릭 선택 → 버튼 수정" 대체안으로 시작 권장** (0장 참고). 시간 남으면 드래그로 고도화 |
| P2 (권장) | F-04 결과물 내보내기 | - | 가장 먼저 잘라도 되는 항목. 복사 버튼 하나만이라도 구현 |

**Cut line 원칙 (4시간 기준):** H2:30에도 통합이 안 됐다면 F-04 → F-03 부분 수정 기능 자체(보여주기만 하고 실제 AI 연동 없이 정적 데모) 순으로 축소. 데모 15분 전 기준으로는 "동작하는 것만 보여준다"가 최우선.

## 3. 팀 커뮤니케이션 계획

- **체크인 주기:** 30분마다 (H0:15, H0:45, H1:15, H1:30, H2:00, H2:30, H3:00, H3:30, H3:50) — 4시간이므로 원래 계획보다 훨씬 촘촘하게
- **채널:** 팀 협의된 채널 사용 — 블로커는 체크인 기다리지 말고 즉시 공유
- **블로커 에스컬레이션:** 15분 이상 막히면 바로 PM에게 공유 → 범위 축소 여부 즉시 판단 (24h 계획 때의 30분 기준보다 짧게 조정)
- **문서 소유:** 이 문서와 [PM 대시보드](대시보드 링크 참고)를 PM이 실시간 업데이트, 팀원은 상태를 대시보드 백로그 카드로 직접 갱신 가능

## 4. 데모 준비 체크리스트

- [ ] 시연 시나리오는 US-01(취준생, 템플릿→초안) 또는 US-02(팀 리더, 회의록 통합) 중 하나를 메인 스토리로 선정
- [ ] 시연용 입력 데이터(예시 자소서 키워드, 예시 회의록 조각) 미리 준비 — 라이브 타이핑으로 시간 낭비 방지
- [ ] 외부 LLM API 응답 지연 대비: 네트워크 안 될 경우 스크린 녹화본 백업
- [ ] "학습 데이터 제외(Zero Data Retention)" 옵션 적용 여부 — 발표 시 보안/프라이버시 포인트로 언급
- [ ] 발표 시간 배분: 문제정의 → 데모(주요 비중) → 기술 하이라이트 → Q&A
- [ ] 발표자/데모 진행자 사전 지정 및 리허설 최소 1회
